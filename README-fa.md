## ⚡ شروع سریع

هر اسکریپت را می‌توانید مستقیم از pool اجرا کنید — بدون clone، بدون دانلود:

```bash
bash <(curl -kLs https://material.bobclub.ir/wp-core.sh)
```

یا کل مخزن را clone کنید:

```bash
git clone git@github.com:babak-majd/boblub-pool.git
cd boblub-pool
chmod +x wp-core.sh
sudo ./wp-core.sh
```

> ### 💾 درباره بکاپ
> **این اسکریپت‌ها بارها روی سرورهای واقعی تست شده‌اند و خودشان هم شبکه ایمنی دارند** — پلاگین قدیمی در `old-<slug>/` نگه داشته می‌شود، هسته قبلی وردپرس در `old-core/`، فایل‌های قبلی سایت در `old-files/`، و دیتابیس Roundcube قبل از حذف در `/root/` بکاپ گرفته می‌شود. یعنی برگشتن به عقب تقریباً همیشه فقط یک گزینه منو فاصله دارد.
>
> با این حال: این اسکریپت‌ها روی فایل‌ها، دیتابیس‌ها و دسترسی‌های زنده سرور می‌نویسند. **اگر با بکاپ گرفتن خیالتان راحت‌تر است، بکاپ بگیرید.** دو دقیقه وقت می‌گیرد و تا حالا هیچ‌وقت کار بدی نبوده.

## 🧩 wp-core.sh

**فایل‌های هسته وردپرس را مدیریت می‌کند** — یعنی `wp-admin/` و `wp-includes/` و فایل‌های PHP ریشه. قالب، پلاگین، فایل‌های آپلودی و دیتابیس شما در عملیات هسته اصلاً دست نمی‌خورند.

**یا داخل `public_html` سایت اجرایش کنید، یا هر جای دیگری اجرا کنید و وقتی پرسید دامنه را بنویسید** — خودش وب‌روت را از cPanel/DirectAdmin پیدا می‌کند.

### اگر وردپرس نصب باشد، این منو را می‌بینید:

| # | گزینه | دقیقاً چه کار می‌کند |
| --- | --- | --- |
| **۱** | تعمیر نسخه فعلی | نسخه نصب‌شده شما (مثلاً 6.9.4) را تشخیص می‌دهد، **همان نسخه** را تازه از wordpress.org دانلود می‌کند و فایل‌های هسته را جایگزین می‌کند. **فایل‌های هک‌شده، خراب یا پاک‌شده هسته را درست می‌کند بدون اینکه نسخه شما عوض شود.** |
| **۲** | به‌روزرسانی به آخرین نسخه | همان کار، ولی با جدیدترین نسخه وردپرس. |
| **۳** | نصب وردپرس 6.9.4 | این نسخه مشخص را نصب می‌کند — برای وقتی که می‌خواهید بعد از یک آپدیت بد، برگردید عقب. |
| **۴** | نصب نسخه دلخواه | نسخه را خودتان تایپ می‌کنید (مثلاً `6.8.3`) و دقیقاً همان نصب می‌شود. |
| **۵** | بازگشت به هسته قبلی | هسته‌ای که دفعه قبل جایگزین شده بود را از `old-core/` برمی‌گرداند. دکمه undo شماست. |
| **۶** | مدیریت کاربران ادمین | زیرمنوی پایین را باز می‌کند. |

گزینه‌های ۱ تا ۴ همگی اول هسته فعلی را داخل **`old-core/`** کپی می‌کنند، پس گزینه ۵ همیشه می‌تواند برش گرداند.

### زیرمنوی مدیریت ادمین (گزینه ۶):

| # | گزینه | چه کار می‌کند |
| --- | --- | --- |
| **۱** | لیست حساب‌های ادمین | همه ادمین‌ها را نشان می‌دهد: نام کاربری، ایمیل، آی‌دی. وقتی سایتی را تحویل می‌گیرید و نمی‌دانید چه کسی داخلش هست، به کار می‌آید. |
| **۲** | تغییر رمز یک ادمین | نام کاربری را می‌دهید و رمز جدید ست می‌شود. **وقتی از سایت قفل شده‌اید، برتان می‌گرداند داخل.** |
| **۳** | ساخت ادمین جدید | یک کاربر ادمین کاملاً جدید با نام کاربری، ایمیل و رمز دلخواه شما می‌سازد. |

این بخش مستقیم با PHP CLI به دیتابیس وصل می‌شود، پس **حتی وقتی wp-admin کاملاً از کار افتاده** هم کار می‌کند.

### اگر وردپرس نصب **نباشد**:

پیشنهاد می‌دهد سایت را از صفر بسازد: فایل‌های موجود به `old-files/` منتقل می‌شوند، دیتابیس از طریق پنل (cPanel/DirectAdmin) ساخته می‌شود — یا خودتان اطلاعات دیتابیس را وارد می‌کنید — وردپرس دانلود می‌شود، `wp-config.php` با salt‌های امنیتی تازه نوشته می‌شود، و مالکیت و دسترسی‌ها درست ست می‌شوند.

```bash
sudo ./wp-core.sh
```

## 🔌 pro-plugin-manager.sh

**جعبه‌ابزار تعمیر برای دو پلاگینی که بیشتر از همه سایت را می‌خوابانند — ووکامرس و المنتور — به‌علاوه search & replace روی کل دیتابیس.**

> ⚠️ این ابزار **برای نصب هر پلاگین دلخواهی نیست.** منوی اصلی دقیقاً همین چهار گزینه است:

| # | گزینه | چه کار می‌کند |
| --- | --- | --- |
| **۱** | WooCommerce Manager | زیرمنوی پلاگین (پایین) را روی `woocommerce` باز می‌کند. |
| **۲** | Elementor Manager | همان زیرمنو، روی `elementor`. |
| **۳** | Search And Replace | جست‌وجو و جایگزینی متن در **تمام جدول‌های** دیتابیس وردپرس. |
| **۴** | نصب آخرین Blue Guard | آخرین نسخه پلاگین امنیتی Blue Guard را نصب می‌کند (نسخه قبلی در `old-blue-guard/` می‌ماند). |

### زیرمنوی پلاگین (گزینه‌های ۱ و ۲):

اول نسخه نصب‌شده فعلی را نشانتان می‌دهد، بعد:

| # | گزینه | چه کار می‌کند |
| --- | --- | --- |
| **۱** | تعمیر نسخه فعلی | دقیقاً همان نسخه‌ای که الان دارید را دوباره دانلود می‌کند و روی پوشه پلاگین می‌نویسد. **پلاگین خراب را بدون تغییر نسخه‌اش درست می‌کند** — این وقتی مهم است که نسخه جدیدتر با قالب شما سازگار نیست. |
| **۲** | به‌روزرسانی به آخرین نسخه | جدیدترین نسخه را از wordpress.org نصب می‌کند. |
| **۳** | نصب نسخه مشخص | نسخه را تایپ می‌کنید (مثلاً `10.9.0`) — برای برگشتن به نسخه‌ای که می‌دانید کار می‌کرد. |
| **۴** | بازگشت به نسخه قبلی | نسخه ذخیره‌شده در `old-<slug>/` از عملیات قبلی را برمی‌گرداند. |

بعد از نصب، می‌پرسد که پلاگین را با wp-cli فعال کند یا نه. دانلود از wordpress.org انجام می‌شود و اگر در دسترس نبود، از یک mirror جایگزین.

### Search & Replace (گزینه ۳)

مقدار قدیمی و مقدار جدید را می‌دهید، بعد انتخاب می‌کنید:

- **۱) Dry run** — فقط می‌شمارد چند ردیف عوض می‌شد. هیچ تغییری نمی‌دهد. **همیشه اول این را بزنید.**
- **۲) جایگزینی واقعی** — واقعاً جایگزین می‌کند.

کاربرد کلاسیکش: انتقال سایت از `http://old-domain.ir` به `https://new-domain.ir`.

> ⚠️ **داده‌های serialize‌شده طولشان ثابت نیست.** وردپرس بعضی تنظیمات (ویجت‌ها، تنظیمات قالب، دیتای صفحه‌سازها) را به‌صورت رشته PHP serialize‌شده ذخیره می‌کند که تعداد کاراکترها داخل خودش نوشته شده. اگر با SQL خام یک رشته با **طول متفاوت** را جایگزین کنید، این داده‌ها خراب می‌شوند. اگر طول مقدار قدیم و جدید فرق دارد و سایت از صفحه‌ساز استفاده می‌کند، به‌جایش از `wp search-replace` در wp-cli استفاده کنید — خود اسکریپت هم همین هشدار را می‌دهد.

```bash
sudo ./pro-plugin-manager.sh
```

## 🎯 plugin-hunter.sh

**آن یک پلاگینی که سایت را کشته پیدا می‌کند.** سناریوی کلاسیک: صفحه سفید یا خطای HTTP 500، هیچ خطایی هم توی لاگ نیست، ۴۰ تا پلاگین نصب است و شما هیچ ایده‌ای ندارید کدامشان مقصر است.

پلاگین‌ها را با تغییر نام پوشه‌شان غیرفعال می‌کند (`plugin-name` ← `plugin-name.off`). وردپرس دیگر آن‌ها را نمی‌بیند. **هیچ چیزی پاک نمی‌شود**، و در پایان همه‌چیز به حالت اول برمی‌گردد — یا اگر لغو کنید، بلافاصله.

### موقع شروع دو چیز از شما می‌پرسد:

**اول — چطور تست شود؟**

| # | حالت | یعنی چه |
| --- | --- | --- |
| **۱** | manual | بعد از غیرفعال شدن هر پلاگین، اسکریپت صبر می‌کند و از **شما** می‌پرسد: «الان درست شد؟» **این را انتخاب کنید** وقتی مشکل چیزی است که فقط آدم می‌فهمد — تسویه‌حساب خراب، به‌هم‌ریختگی ظاهر، صفحه‌ای که نصفه بالا می‌آید. |
| **۲** | automate | خود اسکریپت صفحه اصلی را باز می‌کند و سالم بودن پاسخ را چک می‌کند. سریع‌تر و بدون دخالت شما. این را وقتی بزنید که سایت کاملاً خوابیده (خطای ۵۰۰ / صفحه سفید)، چون این‌ها را یک اسکریپت هم می‌تواند تشخیص دهد. |

**دوم — با چه ترتیبی؟**

| # | استراتژی | یعنی چه |
| --- | --- | --- |
| **۱** | linear | پلاگین‌ها را یکی‌یکی از بالا تست می‌کند. ساده و قابل پیش‌بینی؛ با ۴۰ پلاگین ممکن است تا ۴۰ مرحله طول بکشد. |
| **۲** | binary | **نصف** پلاگین‌ها را یکجا غیرفعال می‌کند، می‌فهمد مقصر در کدام نصف است، بعد باز نصف می‌کند. ۴۰ پلاگین ← حدود ۶ مرحله به‌جای ۴۰. **پیشنهاد می‌شود** — همان نتیجه، با معطلی خیلی کمتر. |

هر کاری که می‌کند در `/var/log/plugin-hunter.log` ثبت می‌شود.

```bash
sudo ./plugin-hunter.sh
# یا مستقیم مسیر را بدهید:
sudo ./plugin-hunter.sh /home/user/domains/site.ir/public_html
```

## 🛡️ perm-patrol.sh

**مالکیت و دسترسی فایل‌های یک اکانت هاست را تعمیر می‌کند** — بعد از یک `chown -R` اشتباه، بعد از مهاجرت از سرور دیگر، یا وقتی چیزی با کاربر `root` آپلود شده. علائمش: «امکان نوشتن در پوشه نیست»، آپلود کار نمی‌کند، آپدیت‌ها شکست می‌خورند، یا `wp-config.php` برای بقیه کاربرهای سرور قابل خواندن است.

پنل را تشخیص می‌دهد (DirectAdmin یا cPanel)، نام کاربری می‌پرسد، وب‌روت‌های آن کاربر را پیدا می‌کند و **فقط داخل همان دایرکتوری خانه** کار می‌کند — از دست زدن به `/` و `/etc` و `/usr` و `/var` و امثالشان قاطعانه امتناع می‌کند.

### سه تا سؤال بله/خیر از شما می‌پرسد، به این ترتیب:

| # | مرحله | چه چیزی را درست می‌کند |
| --- | --- | --- |
| **۱** | بازنشانی مالکیت | همه فایل‌ها به مالکیت کاربر و گروه اصلی اکانت درمی‌آیند. **خطای «permission denied» و آپلود/آپدیت‌های ناموفق را درست می‌کند.** |
| **۲** | اصلاح دسترسی فایل‌های وب | پوشه‌ها ← `755`، فایل‌ها ← `644`. دسترسی استاندارد و امن وب. هم سایت خراب را درست می‌کند، هم فایل‌های خطرناکِ `777` را می‌بندد. |
| **۳** | سخت‌سازی فایل‌های حساس | `wp-config.php` و `.env` و `.my.cnf` و `.htpasswd` ← `600`، یعنی **فقط صاحب اکانت می‌تواند بخواندشان.** رمز دیتابیس و API شما داخل همین فایل‌هاست. |

به هر کدام جدا جواب می‌دهید، پس می‌توانید فقط همان بخشی که لازم دارید را انجام دهید. در پایان گزارش می‌دهد در هر دسته چند فایل را تغییر داده.

### حالت پیش‌نمایش — هیچ چیزی نوشته نمی‌شود:

```bash
sudo ./perm-patrol.sh --dry-run
```

هر تغییری که **قرار بود** بدهد را چاپ می‌کند و بیرون می‌آید. اگر دلشوره دارید اول این را بزنید؛ وقتی مطمئن شدید، بدون فلگ اجرا کنید:

```bash
sudo ./perm-patrol.sh
```

## 📬 fix-roundcube.sh

**وب‌میل خراب Roundcube را روی سرور DirectAdmin از نو می‌سازد.** برای وقتی که وب‌میل خطای دیتابیس می‌دهد، صفحه سفید نشان می‌دهد، یا بعد از یک آپدیت دیگر کسی نمی‌تواند لاگین کند.

منو ندارد — یک کار را از اول تا آخر انجام می‌دهد:

۱. اطلاعات MySQL را از `/usr/local/directadmin/conf/mysql.conf` می‌خواند (لازم نیست رمزی تایپ کنید).
۲. **اول از دیتابیس Roundcube در `/root/` بکاپ می‌گیرد**، با تاریخ و ساعت. اگر بکاپ شکست بخورد هشدار می‌دهد و ادامه می‌دهد — چون نبودِ دیتابیس اغلب دقیقاً همان مشکلی است که دارید درستش می‌کنید.
۳. دیتابیس خراب را drop می‌کند و فایل‌های باقیمانده‌اش را از پوشه داده MySQL پاک می‌کند (این همان کاری است که یک `DROP DATABASE` ساده از پسش برنمی‌آید).
۴. MySQL/MariaDB را دور این پاک‌سازی متوقف و دوباره راه‌اندازی می‌کند.
۵. دستور `da build roundcube` را اجرا می‌کند — دایرکت‌ادمین Roundcube را تمیز از نو می‌سازد.

همه‌چیز در `/tmp/bobclub_log/fix_roundcube.log` لاگ می‌شود.

> 📮 دیتابیس Roundcube فقط **مخاطبین، تنظیمات و ترجیحات پوشه‌ها** را نگه می‌دارد — نه خود ایمیل‌ها را. ایمیل‌های واقعی شما روی دیسک در mail store هستند و **دست نمی‌خورند**.

```bash
sudo ./fix-roundcube.sh
```

## 🔗 thing-to-link.sh

**هر چیزی را تبدیل به لینک دانلود می‌کند.** یک فایل روی سرور دارید و کسی لازمش دارد — مشتری، همکار، یا یک سرور دیگر. به‌جای دادن دسترسی SFTP، یک URL می‌دهید.

اجرایش می‌کنید و فقط **یک** چیز می‌پرسد: مسیر فایل، مسیر پوشه، یا یک URL.

| چه چیزی بدهید… | چه کار می‌کند |
| --- | --- |
| **مسیر فایل** | کپی‌اش می‌کند در `/var/www/html` و لینک عمومی را چاپ می‌کند. |
| **مسیر پوشه** | می‌پرسد `tar.gz` می‌خواهید یا `zip`، فشرده می‌کند، منتشر می‌کند، لینک را می‌دهد. |
